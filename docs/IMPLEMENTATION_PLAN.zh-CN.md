# 垃圾识别与智能回收问答系统实施计划

## 1. 项目概述

本项目建设一个基于 `YOLO26 + Spring Boot + Vue 3 + PostgreSQL` 的垃圾识别与智能回收指导系统，面向图片垃圾识别、分类建议生成和多轮追问场景。

系统主流程如下：

1. 用户上传垃圾图片
2. 后端调用 YOLO26 模型进行检测识别
3. 返回类别、置信度和检测框结果
4. 将识别结果发送给大模型生成垃圾分类与回收建议
5. 用户可基于当前识别会话继续追问

## 2. 建设目标

### 2.1 模型目标

1. 排查本地 `exp.pt` 推理效果弱于官方云端的原因
2. 建立本地可复现推理基线，统一 `imgsz=512`、`conf=0.25`、`iou=0.7`
3. 通过轻量改进提升关键类别识别精度
4. 形成可重复执行的诊断、调参和回归流程

### 2.2 系统目标

1. 实现图片上传、识别、结果展示
2. 实现识别结果传给大模型后生成回收建议
3. 实现基于 `sessionId` 的多轮问答
4. 实现识别记录、会话记录和模型版本追踪
5. 支持后续继续扩展管理端、阈值配置和模型切换功能

## 3. 当前现状判断

已知条件：

1. 已有训练权重 `exp.pt`
2. 文档记录的模型指标并不差，说明模型本身不是完全失效
3. 本地效果不如云端，更可能是推理链路、参数、数据分布或标签边界问题

当前优先判断的根因：

1. 本地与云端使用的模型文件不一致
2. 本地推理参数与云端不一致
3. 本地测试图像与训练/验证集分布差异较大
4. 本地预处理、NMS 或阈值设置不同
5. 类别标签边界模糊，导致混淆严重

## 4. 技术架构

### 4.1 前端

- 技术：`Vue 3 + Tailwind CSS`
- 职责：图片上传、识别结果展示、建议展示、多轮问答、历史记录查看

### 4.2 业务后端

- 技术：`Java 17 + Spring Boot`
- 职责：文件管理、识别流程编排、LLM 调用、会话管理、数据库持久化、统一 API 输出

### 4.3 推理服务

- 技术：`Python + FastAPI + Ultralytics`
- 职责：加载 `exp.pt`、执行推理、返回检测框/类别/置信度/耗时

### 4.4 数据库

- 技术：`PostgreSQL`
- 职责：存储图片、识别结果、识别明细、问答会话、消息、模型版本

## 5. 模型问题排查方案

### 5.1 基线复现

统一本地基线：

1. 模型固定为 `exp.pt`
2. 参数固定为 `imgsz=512 conf=0.25 iou=0.7`
3. 使用固定测试样本集重复验证
4. 记录环境版本：Python、Ultralytics、CUDA/CPU、推理脚本版本

### 5.2 本地与云端差异排查

重点核对：

1. 权重文件是否完全一致
2. 推理参数是否一致
3. 图片输入尺寸和缩放方式是否一致
4. 是否启用了不同的后处理规则
5. 类别名称映射是否一致

### 5.3 错误样本分析

按以下类别整理样本：

1. 漏检
2. 误检
3. 类别混淆
4. 低置信度
5. 复杂背景
6. 遮挡样本

### 5.4 诊断输出

每次基线运行至少记录：

1. 无检测比例
2. top-1 类别分布
3. top-1 平均置信度
4. 输出目录与时间戳

## 6. 精度优化路线

### 6.1 第一阶段：不改网络结构的优化

1. 每类阈值调优
2. 标签清洗
3. 困难样本回放训练
4. 长尾类别重采样
5. 增加真实拍摄场景样本

### 6.2 第二阶段：轻量算法改进

1. 引入 `Focal Loss`
2. 增加类别权重
3. 比较 `yolo26n` 与更大规模模型
4. 对低置信度场景增加二次判断机制

### 6.3 实验要求

建议至少保留以下实验组：

1. Baseline
2. Baseline + 阈值调优
3. Baseline + 标签清洗/困难样本回放
4. Baseline + Focal Loss
5. Baseline + Focal Loss + 类别权重
